# sqlrs diff

English: [sqlrs-diff.md](sqlrs-diff.md)

## Обзор

**Статус: дизайн / будущее.** Команда входит в набор git-aware passive, описанный в
[`docs/architecture/git-aware-passive.RU.md`](../architecture/git-aware-passive.RU.md).
В текущем MVP CLI не реализована.

`sqlrs diff` показывает **что изменилось в контексте prepare** между двумя входами:
двумя Git ref'ами (например, база и PR) или двумя локальными путями. В отличие от
обычного `git diff`, фокус на **наборе файлов миграций/скриптов**, который будет
использоваться при `sqlrs run --prepare <path>`, и выдаёт структурированный отчёт
(Added / Modified / Removed) с опциональными фрагментами содержимого и краткой сводкой.

Примеры использования:

- Просмотр изменений миграций между ветками или тегами перед запуском.
- Сравнение двух локальных деревьев миграций без Git.
- CI: машинно-читаемый diff для согласования или уведомлений.

---

## Терминология

- **Контекст prepare** — множество файлов (миграции, скрипты) по пути, который
  передаётся в `sqlrs run --prepare <path>` (или будет использоваться с `--ref` в
  будущем). То же понятие задаёт «откуда» и «куда» для diff.
- **Changeset** — одно файловое изменение: Added, Modified или Removed.

---

## Синтаксис команды

Два режима взаимоисключающие: **по ref'ам** (Git) или **по путям** (только локально).

### Режим 1: Diff по Git ref'ам и контексту prepare

```text
sqlrs diff --from-ref <refA> --to-ref <refB> --prepare <path> [OPTIONS]
```

- `<refA>`, `<refB>` — Git ref'ы: `HEAD`, `origin/main`, хеш коммита, тег (например
  `v1.2.3`) или `refs/pull/123/head`, если доступно локально.
- `<path>` — путь к миграциям/скриптам **относительно корня репозитория** (файл или
  каталог). Определяет, какие файлы участвуют в diff (та же семантика, что у
  `sqlrs run --prepare` в будущем).

### Режим 2: Diff двух локальных наборов (без Git)

```text
sqlrs diff --from-path <pathA> --to-path <pathB> [OPTIONS]
```

- `<pathA>`, `<pathB>` — локальные пути (файлы или каталоги) для сравнения.

Нельзя смешивать ref'ы и пути: например, `--from-ref` и `--from-path` одновременно
указывать нельзя.

## Опции

| Опция | По умолчанию | Описание |
|-------|--------------|----------|
| `--format` | `text` | Формат вывода: `text` или `json`. |
| `--include-content` | `false` | Включать фрагменты содержимого для Modified (и опционально Added/Removed) в отчёт. |
| `--limit` | (нет) | Максимальное число записей об изменениях в списке (для длинных diff). |
| `--ref-mode` | `blob` | (Только режим ref.) Как читать файлы из Git: `blob` (только объекты Git) или `worktree` (временный worktree). Та же семантика, что у `sqlrs run --ref` (см. git-aware-passive). |
| `--workspace` | (конфиг/текущая директория) | Рабочий каталог (корень репо и т.д.), как для `sqlrs run`. |

## Вывод

### Текстовый формат (`--format text`)

- Краткий заголовок с сравниваемыми контекстами (from/to ref'ы или пути).
- Секции: **Added**, **Modified**, **Removed** с перечислением затронутых путей (и
  опционально идентификаторов вроде хеша/размера).
- Для **Modified** при `--include-content`: фрагменты в стиле unified diff или
  короткие отрывки.
- Строка **summary**: количество добавленных, изменённых и удалённых файлов (и
  опционально строк).
- Опционально строка **DB impact**: `yes` | `no` | `unknown` (эвристика: только
  комментарии/пробелы → `no`; иначе `unknown` или `yes`, если семантика миграций
  распознаётся).

### JSON-формат (`--format json`)

Один JSON-объект со стабильной структурой для скриптов и CI:

- `from`, `to` — идентификаторы контекстов «откуда» и «куда» (ref или путь).
- `added` — массив записей (путь и опционально содержимое/хеш).
- `modified` — массив записей (путь, опционально старый/новый хеш, опционально содержимое).
- `removed` — массив записей (путь и опционально содержимое).
- `summary` — объект с счётчиками (например `added_count`, `modified_count`, `removed_count`).
- `db_impact` — опциональная строка: `yes` | `no` | `unknown`.

При заданном `--limit` массивы могут быть обрезаны; summary при этом по возможности
отражает полные счётчики.

## Валидация и ошибки

- **Не git-репозиторий** — при использовании `--from-ref`/`--to-ref` текущий
  workspace должен находиться внутри Git-репо. Иначе CLI завершается с ошибкой и
  предлагает использовать `--from-path`/`--to-path`.
- **Невалидный или неразрешаемый ref** — если ref не разрешается в коммит/дерево,
  команда завершается с понятным сообщением.
- **Конфликт режимов** — если заданы и ref-, и path-опции (например `--from-ref` и
  `--from-path`), команда завершается с ошибкой.
- **Отсутствующий путь** — в режиме по путям оба `--from-path` и `--to-path` должны
  существовать (файл или каталог).

## Примеры

### Diff между веткой и базой (режим ref)

```bash
sqlrs diff --from-ref origin/main --to-ref HEAD --prepare migrations/
```

### Diff между двумя тегами

```bash
sqlrs diff --from-ref v1.0 --to-ref v2.0 --prepare db/scripts
```

### JSON-вывод с содержимым и лимитом

```bash
sqlrs diff --from-ref main --to-ref feature/schema --prepare migrations/ \
  --format json --include-content --limit 50
```

### Diff двух локальных каталогов (без Git)

```bash
sqlrs diff --from-path ./migrations-v1 --to-path ./migrations-v2
```

```bash
sqlrs diff --from-path ./old.sql --to-path ./new.sql --format json
```

## Заметки для реализаторов

Алгоритм (из дизайна):

1. Загрузить файлы контекста из `from-ref`/`to-ref` или `from-path`/`to-path` (blob
   или worktree согласно `--ref-mode` в режиме ref).
2. Нормализовать входной набор (например, каталог → упорядоченный набор файлов).
3. Вычислить хеши и сравнить: Added/Removed по пути, Modified по хешу.
4. Опционально построить семантическую подсказку (например, только комментарии/пробелы
   → низкое или нулевое влияние на БД).
5. Выдать отчёт в запрошенном формате.

Команда не запускает engine, не выполняет миграции и не обращается к кэшу; только
сравнивает наборы файлов и содержимое.
