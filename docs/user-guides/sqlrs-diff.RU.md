# sqlrs diff

English: [sqlrs-diff.md](sqlrs-diff.md)

## Обзор

**Статус: дизайн / будущее.** Команда входит в набор git-aware passive, описанный в
[`docs/architecture/git-aware-passive.RU.md`](../architecture/git-aware-passive.RU.md).
В текущем MVP CLI не реализована.

`sqlrs diff` — **meta-команда**: она оборачивает одну существующую content-aware
команду sqlrs, вычисляет её в двух контекстах и выводит разницу. Цель — сохранить
синтаксическую близость к основным командам sqlrs, а не вводить отдельный DSL для
diff.

**Составные команды diff.** Для максимальной совместимости с основным CLI `diff`
оформлен как **группа составных команд**: берётся тот же глагол, что и для
обычного запуска, а между `sqlrs` и ним вставляется область diff. Три формы и что
они сравнивают:

| Форма | Сравнивает |
|-------|------------|
| `sqlrs diff ... plan ...` | **Разницу в планах задач** по подготовке экземпляра (порядок шагов, хеши задач, кэшируемость). |
| `sqlrs diff ... prepare ...` | **Разницу в телах задач** по подготовке экземпляра (разрешённые файлы, граф include, нормализованные единицы контента). |
| `sqlrs diff ... run ...` | **Разницу в телах задач** по выполнению запросов (только файловые входы run). |

То есть: `diff ... plan` — «как меняется *план*?»; `diff ... prepare` — «как
меняются *входы/тела prepare*?»; `diff ... run` — «как меняются *входы run*?».
Та же командная строка, что и для `plan` / `prepare` / `run`, повторно используется,
с вставленной только областью diff.

Примеры использования:

- Посмотреть, как результат `plan:*` меняется между ветками.
- Посмотреть, как граф входных файлов или тела задач `prepare:*` меняются между
  ревизиями.
- Сравнить два локальных дерева с той же семантикой команд sqlrs, без Git.

## Синтаксис команды

```text
sqlrs [global-options] diff (--from-ref <refA> --to-ref <refB> | --from-path <pathA> --to-path <pathB>) <sqlrs-command> [command-args...]
```

Где:

- `diff` задаёт только **область сравнения**.
- `<sqlrs-command>` — одна оборачиваемая команда sqlrs; она должна использовать
  **тот же синтаксис**, что и в основном CLI: тот же подкоманда, те же опции, тот
  же `--` и хвостовые аргументы. Актуальный синтаксис каждой команды см. в
  [`sqlrs-plan.md`](sqlrs-plan.md), [`sqlrs-prepare.md`](sqlrs-prepare.md) и
  [`sqlrs-run.md`](sqlrs-run.md) (и в вариантных гайдах).
- **Глобальные опции** действуют как при любом вызове sqlrs: `-v` / `--verbose`
  включает более подробное протоколирование в stdout/stderr в процессе diff;
  `--output human|json` выбирает формат вывода (отдельного `--format` у diff нет).

Примеры оборачиваемых команд:

- `plan:psql`
- `plan:lb`
- `prepare chinook`
- `prepare:psql`
- `prepare:lb`
- `prepare chinook run:psql -- -f queries.sql`
- `prepare:psql -- -f prepare.sql run smoke`

В первом срезе не делаем:

- отдельные флаги вывода вроде `--format` для diff;
- изменение синтаксиса оборачиваемой команды.

## Выбор области (scope)

Нужно выбрать ровно один режим области.

### Режим 1: Сравнение двух Git ref'ов

```text
sqlrs diff --from-ref <refA> --to-ref <refB> <sqlrs-command> [command-args...]
```

- `<refA>`, `<refB>` — например `HEAD`, `origin/main`, хеш коммита, тег или любой
  локально разрешаемый Git ref.
- Оборачиваемая команда вычисляется отдельно на каждой ревизии.

### Режим 2: Сравнение двух локальных путей

```text
sqlrs diff --from-path <pathA> --to-path <pathB> <sqlrs-command> [command-args...]
```

- `<pathA>`, `<pathB>` задают левый/правый локальный контекст.
- Оборачиваемая команда вычисляется отдельно для каждого дерева.

Нельзя смешивать ref- и path-опции области.

## Семантика оборачиваемой команды

`sqlrs diff` не выполняет оборачиваемую команду в обычном смысле. Вместо этого он
строит **производное представление**, важное для этой команды, и сравнивает две
стороны.

**Зависимые от ревизии и прочие аргументы:** Сравниваются только те входы, которые
зависят от репозитория или контекста путей (например, файлы из `-f` или граф
include). Аргументы, не зависящие от ревизии (например, `-c 'SELECT 1'` в psql
или `--image postgres:17`), на обеих сторонах одинаковы; diff для них отдельной
семантики не вводит. Если команда смешивает файловые и инлайн-входы, diff'ится
только файловая часть; при отсутствии чего-либо зависящего от ревизии реализация
может выдать предупреждение или отклонить вызов.

### `plan:*`

Сравнивается производный план задач:

- порядок задач,
- тела задач / хеши,
- входы, влияющие на кэшируемость,
- разрешённый граф файлов, влияющий на планирование.

### `prepare:*`

Сравнивается payload подготовки:

- разрешённые входные файлы,
- развёрнутый граф include,
- нормализованные тела задач / единицы, производные от контента.

### `run:*` (будущее / ограниченно)

`run:*` — будущее расширение; осмыслено только для **файловых входов**. Аргументы,
не зависящие от ревизии (например, `-c 'select 1'` в `run:psql`), не диффятся —
ref в Git не меняет этот payload. Вызовы только с инлайном могут отклоняться или
давать пустой diff. Тот же принцип для других видов: сравниваются только входы из
файлов (или из разрешённого графа файлов).

### Составная форма `prepare ... run`

`sqlrs diff` также принимает ту же двухстадийную составную форму, что и основной
CLI:

- `prepare <prepare-ref> run <run-ref>`
- `prepare <prepare-ref> run:<kind> ...`
- `prepare:<kind> ... run <run-ref>`
- `prepare:<kind> ... run:<kind> ...`

Правила:

- обёрнутые стадии `prepare` и `run` разрешаются на левой и правой стороне по тем
  же правилам, что и в основном CLI;
- стадии на alias-файлах отдельно находят свои `*.prep.s9s.yaml` и `*.run.s9s.yaml`
  на каждой стороне;
- относительные alias ref не переразрешаются относительно workspace root и в diff
  тоже: каждая сторона считает их от своего effective working directory;
- файловые пути, прочитанные из alias-файла, считаются относительно директории
  этого alias-файла на соответствующей стороне;
- diff показывает фазу `prepare` и фазу `run` отдельно;
- если одна из фаз не имеет revision-dependent payload, она сообщается как пустая
  или non-diffable, не ломая вторую фазу.

Для режимов области effective working directory таков:

- в path-mode: корень стороны, указанный через `--from-path` или `--to-path`;
- в ref-mode: текущий рабочий каталог вызывающего процесса, спроецированный в
  дерево соответствующего ref.

### Зависимое от ревизии формирование списка файлов

**Набор файлов**, участвующих в оборачиваемой команде, может различаться между
двумя сторонами. Например, в старой ревизии скрипт мог подключать два файла через
`\i`, в новой — три; или включённый файл переименован или удалён. Реализация diff
должна выполнять **полное обнаружение файлов и развёртку include в контексте
каждой стороны** и сообщать:

- добавленные или удалённые файлы в разрешённом графе,
- изменения содержимого файлов, присутствующих в обоих графах.

То есть «какие файлы важны» выводится отдельно для каждой ревизии, а не предполагается
одинаковым с обеих сторон.

### Построение списка файлов по kind

С точки зрения diff команды (plan, prepare, run) отличаются в основном тем, **как
строится список файлов** для данного kind. Реализация должна уметь строить
**замыкание файловых входов** для каждого prepare kind и run kind по тем же
правилам, что и основной CLI.

| Kind | Точка входа | Правило замыкания |
|------|-------------|-------------------|
| **prepare:psql** / plan:psql | `-f <файл>` (и `-f -` со stdin) | Замыкание по `\i`, `\ir`, `\include`, `\include_relative` (и другим директивам include, которые раскрывает движок). Старт от файла(ов) из `-f`; рекурсивно добавлять все файлы, на которые они ссылаются. |
| **prepare:lb** / plan:lb | `--changelog-file <path>` | Замыкание по графу changelog: старт от файла changelog; добавить все файлы, на которые он ссылается (include, includeAll и т.д.). Граф задаёт Liquibase; diff использует то же разрешение, чтобы набор файлов совпадал с тем, что использует prepare/plan. |
| **run:psql** (будущее) | `-f <файл>` (только файловые входы) | Как у prepare:psql: замыкание по `\i` / `\include` от каждого `-f`. Инлайн `-c` в список файлов не входит. |
| **run:*** (другие kinds, будущее) | Зависит от kind | Каждый run kind задаёт, какие аргументы считаются файловыми; список файлов строится по ним (например, путь к скрипту, конфигу) и по их kind-специфичным include. |

После построения списка файлов (замыкания) для каждой стороны diff сравнивает
два набора и их содержимое (Added / Modified / Removed). Выбор plan vs prepare
vs run влияет только на *что* выводится из этих файлов (план задач, тела задач
или payload run); главное требование к реализации — механизм построения списка
файлов для каждого kind.

## Опции, специфичные для diff

| Опция | Описание |
|-------|----------|
| `--from-ref <ref>` | Левая Git-ревизия. |
| `--to-ref <ref>` | Правая Git-ревизия. |
| `--from-path <path>` | Левый локальный контекст. |
| `--to-path <path>` | Правый локальный контекст. |
| `--ref-mode blob\|worktree` | Только в режиме ref. Как загружать файлы из Git. |
| `--include-content` | Включать фрагменты содержимого в human/json-вывод. |
| `--limit <n>` | Ограничить число записей в списке для очень больших diff. |

Вывод и уровень детализации — **глобальные**, не специфичные для diff:

- `--output human|json` — глобальный выбор формата вывода (как у `ls`, `plan` и т.д.).
- `-v` / `--verbose` — глобальный флаг подробности: при установке sqlrs выводит
  больше деталей о процессе diff (разрешение области, обнаружение файлов) в
  stdout/stderr.

## Вывод

### Человеко-читаемый вывод (`--output human`)

- Заголовок: область сравнения и оборачиваемая команда.
- Секции Added / Modified / Removed.
- Опциональные фрагменты содержимого при `--include-content`.
- Краткие счётчики.
- Опциональная подсказка вроде `db_impact=yes|no|unknown`, если доступно.

### JSON-вывод (`--output json`)

Один JSON-объект со стабильными полями верхнего уровня:

- `scope`, `command`, `added`, `modified`, `removed`, `summary`, `db_impact` (опц.).

При использовании `--limit` списки могут обрезаться; счётчики в summary по
возможности отражают полный diff.

## Валидация и ошибки

- **Нет или конфликт области** — должна быть выбрана ровно одна из режимов (ref
  или path).
- **Недопустимая оборачиваемая команда** — diff требует одну команду sqlrs.
- **Неподдерживаемая форма** — вложенный composite `prepare ... run` в первом
  срезе отклоняется.
- **Ошибка разрешения ref** — один из ref'ов не разрешается локально.
- **Не Git-репозиторий** — режим ref требует контекста Git-репо.
- **Нет payload, зависящего от ревизии** — для будущей поддержки `run:*` вызовы
  только с инлайном могут отклоняться как не-diffable.

## Примеры

### Сравнение двух ref'ов с `plan:psql`

```bash
sqlrs diff --from-ref origin/main --to-ref HEAD plan:psql -- -f ./prepare.sql
```

### Сравнение двух ref'ов с `prepare:lb`

```bash
sqlrs diff --from-ref origin/main --to-ref HEAD prepare:lb -- \
  update --changelog-file db/changelog.xml
```

### Сравнение двух локальных деревьев с `prepare:psql`

```bash
sqlrs --output json diff --from-path ./left --to-path ./right prepare:psql -- -f ./prepare.sql
```

## Заметки для реализаторов

Ключевое требование — **построение списка файлов (замыкания) для каждого prepare
kind и run kind** по правилам из таблицы выше; diff затем сравнивает эти списки и
их содержимое. Остальное — общая обвязка.

1. Сначала разбирать глобальные флаги так же, как при обычном вызове sqlrs.
2. Разобрать блок области `diff`.
3. Разобрать оборачиваемую команду по **той же грамматике и правилам валидации**,
   что и в основном CLI (для сохранения совместимости синтаксиса).
4. Для каждой стороны (from-ref/to-ref или from-path/to-path) **построить список
   файлов** для данного kind: разрешить ref или путь, затем применить правило
   замыкания этого kind (например, `\i`/`\include` для psql, граф changelog для lb).
   Набор файлов и содержимое могут различаться.
5. При необходимости построить производное представление (план задач, payload
   prepare, файловые входы run) для каждой стороны, если этого требует формат
   вывода.
6. Сравнить два набора файлов (и при необходимости производные представления) и
   классифицировать Added / Modified / Removed.
7. Вывести результат в формате human или JSON согласно глобальному `--output`.

Команда в первом срезе дизайна не запускает engine и не выполняет SQL; она
сравнивает значимые для sqlrs наборы файлов и их содержимое. Совместимость с
основным CLI (синтаксис, глобальные `-v`, `--output`) сделана намеренно, чтобы
пользователь мог подставлять ту же командную строку, что и для `plan` или
`prepare`, лишь вставляя область `diff`.

**Структура компонентов и поток вызовов** (кто строит списки файлов, кто кого
вызывает) описаны в [`docs/architecture/diff-component-structure.RU.md`](../architecture/diff-component-structure.RU.md).
